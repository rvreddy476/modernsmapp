1. System Architecture Overview
The Postbook platform is a social networking ecosystem consisting of three consumer-facing applications (Postbook, Postgram, PostTube) powered by 8 backend microservices and 5 domain-grouped databases. This document is the single source of truth for all services, schemas, APIs, and inter-service communication patterns.

1.2 Database Domain Grouping
Services that share closely related data are grouped onto the same PostgreSQL instance. Each group is a separate PostgreSQL database (not schema) for clean isolation with the option to split onto separate servers later.

Database
Services
Tables
Rationale
identity_db
Auth + Profile
users, sessions, profiles, profile_about, profile_links, channels, business_pages, privacy, reputation, preferences
Auth and Profile always need each other. Profile reads user_id from users on every request. Keeping them together avoids cross-DB joins.
social_db
Chat + Notification
conversations, messages, message_reads, notification_templates, notifications, notification_preferences, user_devices
Chat generates notifications. Co-locating avoids async delays for read receipts and typing indicators.
content_db
Content + Media
posts, comments, reactions, stories, videos, media_assets, transcoding_jobs, playlists
Content and media are tightly coupled. Every post/video references media assets. Same transaction boundary.
search_db
Search
Elasticsearch indices (not PostgreSQL)
Separate engine. Synced from other DBs via events.
monetization_db
Monetization
wallets, transactions, subscriptions, payouts, creator_monetization, tax_info
Financial data needs strict isolation for security, auditing, and compliance.

1.3 Inter-Service Communication
Services communicate via two patterns depending on the use case:

Pattern
When to Use
Technology
Examples
Synchronous REST
Request needs an immediate response; user is waiting
HTTP/REST via Axios (existing interceptor pattern)
Profile Service calls Auth Service to validate JWT; Content Service calls Profile to get author info
Async Event Bus
Fire-and-forget; eventual consistency is fine; fan-out to multiple consumers
Redis Streams or RabbitMQ
New post created → Notification Service sends push; User updates profile → Search Service reindexes

1.5 API Gateway Pattern
All client requests go through a single API Gateway. The gateway handles: SSL termination, rate limiting (global), JWT validation (pass token to downstream services), request routing based on URL prefix, and CORS headers. The gateway does NOT do business logic.

2. Identity Database (Auth + Profile Services)
This is the most critical database. It stores user authentication data and all profile information. Both the Auth Service and Profile Service connect to this database. The Auth Service owns the users and sessions tables; the Profile Service owns everything else.

⚠️  CRITICAL: All table and column names MUST use lowercase snake_case. Never use quoted identifiers. This prevents the PostgreSQL case-sensitivity bugs previously encountered with 'Id' vs 'id'.

3. Social Database (Chat + Notification Services)
This database handles all real-time communication and notification delivery. The Chat Service owns conversations and messages; the Notification Service owns notification tables.

ℹ️  NOTE: Chat and Notification services only store user_id (UUID) references. They do NOT store usernames, display names, or avatars. These are fetched from Profile Service via REST call (with Redis caching, TTL 5 min) when rendering UI.

5. Search Engine (Elasticsearch)
The Search Service does NOT use PostgreSQL. It maintains Elasticsearch indices that are synced from other databases via the event bus. The Search Service is read-heavy and never the source of truth.

5.1 Index: profiles
Synced from identity_db on profile.updated and user.registered events.
{ "username": "text + keyword", "display_name": "text",
  "bio": "text", "category": "keyword",
  "interests": "keyword[]", "location_city": "text",
  "location_country": "keyword", "is_verified": "boolean",
  "follower_count": "long", "account_type": "keyword",
  "created_at": "date" }

5.2 Index: posts
Synced from content_db on post.created, post.updated, post.deleted events.
{ "author_id": "keyword", "author_username": "keyword",
  "content": "text", "hashtags": "keyword[]",
  "app_origin": "keyword", "post_type": "keyword",
  "like_count": "long", "comment_count": "long",
  "visibility": "keyword", "created_at": "date" }

5.3 Index: videos
{ "title": "text", "description": "text",
  "channel_handle": "keyword", "author_id": "keyword",
  "category": "keyword", "tags": "keyword[]",
  "view_count": "long", "like_count": "long",
  "duration_seconds": "integer", "status": "keyword",
  "created_at": "date" }

5.4 Index: channels
{ "handle": "keyword + text", "name": "text",
  "description": "text", "category": "keyword",
  "subscriber_count": "long", "country": "keyword",
  "language": "keyword", "is_verified": "boolean" }

5.5 Index: business_pages
{ "page_handle": "keyword", "page_name": "text",
  "category": "keyword", "description": "text",
  "location": "geo_point", "avg_rating": "float",
  "review_count": "integer", "is_verified": "boolean" }

5.6 Sync Strategy
Event-driven sync: each event triggers an index update in near-real-time (< 2 seconds).
Full reindex: scheduled weekly to catch any missed events. Run during low-traffic hours.
Delete sync: on user.deleted or post.deleted events, remove from index immediately.
Bulk operations: use Elasticsearch bulk API for batch syncs (> 100 documents).

6. Monetization Database
Strictly isolated for security and compliance. Contains all financial data. This database has additional security: encrypted at rest, restricted access (only Monetization Service), full audit logging on every write.

🔒  SECURITY: This database must have: encryption at rest (AES-256), column-level encryption for payout details and tax info, full audit trail on every INSERT/UPDATE/DELETE, separate database credentials from all other services, no direct access from any other service – only via Monetization Service API.

9. Privacy Engine Logic
The Privacy Engine is a shared middleware used by the Profile Service. Every API response returning profile data passes through it. Other services (Content, Chat) call Profile Service for privacy checks rather than implementing their own.

9.1 Relationship Resolution
Evaluate in order: (1) Owner? Return all. (2) Blocked? Return 404. (3) In custom list? Apply list visibility. (4) Friend? Apply friends and above. (5) Follower? Apply followers and above. (6) Otherwise? Public only.

9.2 Implementation
async function filterByPrivacy(profileData, viewerId, ownerId) {
  if (viewerId === ownerId) return profileData;
  if (await isBlocked(viewerId, ownerId)) throw NotFound();
  const relationship = await getRelationship(viewerId, ownerId);
  const privacySettings = await getPrivacySettings(ownerId);
  if (privacySettings.temp_hide_until > NOW && relationship !== 'owner')
    throw NotFound('Profile temporarily unavailable');
  return filterFields(profileData, relationship, privacySettings);
}

9.3 Cross-Service Privacy Checks
When the Content Service needs to check if User A can see User B's post, it calls:
GET /api/v1/profiles/internal/can-view?viewer=uuid&target=uuid&level=friends
This internal endpoint is not exposed through the API Gateway. It is service-to-service only, authenticated via internal API key.

10. Unique / Differentiating Features

10.1 Profile Audio/Video Intro
What: 10-second clip pinned to profile header. Auto-plays muted on visit.
Storage: intro_media_url in profiles table. Upload via Media Service. Max 10s, 5MB.
Validation: Media Service runs ffprobe to verify duration ≤ 10s. Rejects longer clips.
Frontend: Small play button on avatar. Tap to expand mini-player. Video: muted + captions. Audio: waveform animation.

10.2 Built-in Link Tree with Analytics
What: Unlimited categorized links with free click analytics. Kills Linktree dependency.
Click Tracking: POST /api/v1/links/:id/click is fire-and-forget. Atomic: UPDATE profile_links SET click_count = click_count + 1. Rate limit: 10 clicks/link/min/IP.
Analytics: Clicks per link, daily/weekly/monthly trends, top links. Cached in Redis with 1h TTL.

10.3 Taste Compatibility Score
What: 'You both like N things' shown when viewing another profile.
Algorithm: Intersect interests, hobbies, favorites across both users. Display matches. Cache 24h in Redis.

10.4 Flexible Life Timeline
What: Unified work/education/achievement timeline instead of rigid LinkedIn-style sections.
Display: Vertical timeline on Postbook. Compact cards on Postgram. About tab on PostTube.

10.5 Living Status / Mood
What: Ephemeral status on profile. Auto-expires. Examples: 'Currently into: Rust' / 'Available for: collabs'.
Cleanup: Cron job every 5 min: UPDATE profiles SET status_text=NULL, status_emoji=NULL WHERE status_expires_at < NOW().

10.6 Portable Reputation System
Trust Score: 0–1.0 computed from: account age (0.2), verification (0.2), endorsements (0.2), behavior (0.2), content quality (0.2). Show as tier badge, not raw score.
Cross-Platform Proof: Link external accounts. User posts verification code on other platform. We verify via API.

10.7 Temporary Profile Mode
What: Soft deactivation for 1–90 days. Profile returns 404 to others. All data persists. Early unhide available.

13. Critical Rules & Constraints
The coding agent MUST follow every rule below. Violations cause bugs, security holes, or data corruption.

13.1 PostgreSQL Naming
ALL table names: lowercase snake_case (profile_about, NOT ProfileAbout)
ALL column names: lowercase snake_case (user_id, NOT userId or UserId)
NEVER use quoted identifiers. If you need quotes, the name is wrong.
This rule prevents case-sensitivity bugs with foreign keys (previously encountered 'Id' vs 'id' issue).

13.2 Cross-Database Rules
NEVER create foreign keys across databases. Use UUIDs as references and validate at the app layer.
Store only user_id (UUID) in social_db and content_db. Fetch profile data via REST call to Profile Service.
Cache cross-service data in Redis: profile info (TTL 5 min), privacy checks (TTL 1 min).
All inter-service REST calls use internal API keys, not user JWTs.
Event bus messages must be idempotent. Consumers may receive duplicates.

13.3 Security
NEVER store raw passwords. Use bcrypt (cost 12+) or argon2id.
NEVER return password_hash, two_factor_secret, details_encrypted, or tax_data_encrypted in any API response.
Monetization DB: encrypted at rest, column-level encryption for payout details and tax info.
Rate limits: login 5/min/IP, link clicks 10/min/IP, profile views 60/min/IP, uploads 10/min/user, search 30/min/user.
JWT access tokens: 15 min expiry. Refresh tokens: 7 day expiry. Existing interceptor pattern applies.
Validate all uploads server-side: file type whitelist, max sizes, image dimensions, video duration.
WebSocket connections must authenticate on connect with valid JWT.

13.4 Data Integrity
Every FK within same DB: ON DELETE CASCADE or ON DELETE SET NULL (decide per relationship).
Cross-DB references: handle cleanup via user.deleted event (all services consume it).
Use CHECK constraints at database level for all ENUMs.
All timestamps: TIMESTAMPTZ (with timezone), never TIMESTAMP.
All primary keys: UUID via gen_random_uuid(), never auto-increment (except audit_log).
Username uniqueness: DB-level UNIQUE index + app-level case-insensitive check (LOWER(username)).

13.5 API Design
Standard envelope: { success, data, error: {code, message}, meta: {page, limit, total, has_next} }
HTTP status codes: 200 OK, 201 Created, 204 Deleted, 400 Validation, 401 Unauth, 403 Forbidden, 404 Not Found, 409 Conflict, 429 Rate Limit.
Pagination default: page=1, limit=20, max=100. Cursor-based for messages (use created_at).
PATCH = partial update. Only provided fields are modified.
File uploads: multipart/form-data. Never base64 in JSON body.
Internal service-to-service endpoints: prefix with /internal/, not exposed via gateway.

13.6 Performance Targets
Profile header: < 50ms (cached), < 200ms (uncached).
Feed generation: < 300ms for first page.
Search results: < 200ms.
Message send: < 100ms to acknowledge, delivered via WebSocket within 500ms.
Use Redis for: sessions, profile cache (5 min), compatibility scores (24h), rate limiting, typing indicators (3s), unread counts.
Denormalize counters (follower_count, like_count, etc.). Update via background jobs, not inline.
Link click counting: atomic UPDATE or Redis counter with periodic DB flush.
